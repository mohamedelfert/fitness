# E1.9 — SaaS billing (B2C) — Design

> **Status:** approved design, pre-implementation · **Date:** 2026-06-29 · **Epic:** E1.9 (PHASE_PLAN) · **FRs:** FR-SAS-002, FR-SAS-003, FR-SAS-004 (top-up).

## 1. Goal

Let a Person subscribe to a SaaS plan (free / premium), with trials, coupons, upgrade/downgrade (proration), cancellation, and AICredit top-ups — and have plan membership **grant AICredits** and **gate features**. This replaces the pre-billing free-grant stopgap (the `OnboardingCompleted → GrantFreeAiCredits` listener) with real plan-driven grants.

## 2. Confirmed scope decisions

- **Full lifecycle**, built behind a **fake `PaymentGateway` seam** — mirrors the AI Brain's `LlmGateway`/ADR-004 pattern. The whole mechanism (subscribe, webhook reconciliation, proration, top-up) is built + TDD-tested now against a fake gateway; the real provider adapter + concrete prices land later.
- **In scope (user-confirmed):** server-side entitlement enforcement, coupons & redemptions, **and** proration on upgrade/downgrade — i.e. all of FR-SAS-003.
- **Async webhook-driven activation** — model the real PSP flow (redirect → webhook), not a synchronous shortcut. Free/trial plans activate immediately.

## 3. External blockers (do NOT block this work; drop in behind the seam)

- **Q2 — pricing.** Plan `price_micros` are placeholders until product sets them. Modelling is unaffected.
- **Q3 — MENA PSP** (Paymob / HyperPay / Tap). The real `PaymentGateway` adapter + webhook signature verification land with Q3. Until then `UnconfiguredPaymentGateway` is bound in prod (throws), `FakePaymentGateway` in tests/local.

## 4. Architecture

### 4.1 Module placement
New **`Billing`** module (Plane A): plans, subscriptions, coupons, the payment gateway seam, webhook handling, entitlements. The **AICredit wallet/ledger/meter stay in `AiOrchestration`** — Billing grants credits through the existing `AiCreditMeter::grant()` (the cross-module pattern the `GrantFreeAiCredits` listener already uses). Dependency direction: `Billing → AiOrchestration` (grant credits), `Identity` (Person, OnboardingCompleted). No cycle.

### 4.2 Payment seam (mirrors `LlmGateway`, ADR-004)
```
interface PaymentGateway {
    createCheckout(CheckoutIntent $intent): CheckoutSession   // returns redirect_url + psp_ref
    verifyWebhook(string $payload, string $signature): WebhookEvent   // throws on bad signature
    refund(string $pspRef, int $amountMicros): void
}
```
- **Default prod binding:** `UnconfiguredPaymentGateway` — every method throws (like `UnconfiguredLlmGateway`). Forces the real adapter at Q3.
- **Test/local binding:** `FakePaymentGateway` — deterministic checkout (`psp_ref` = generated), and a test helper to synthesise webhook events so the async flow is exercised end-to-end without a network.
- Real adapter(s) (`PaymobGateway`, …) bind in `BillingServiceProvider` when Q3 lands.

## 5. Data model (Plane A, `Billing` module)

- **`plans`** — `id, audience enum[b2c|coach|gym], code (unique slug e.g. 'free'|'premium'), name, price_micros, currency, interval enum[month|year]|null, ai_credit_grant int, limits_json, features_json, active bool, timestamps`. Seeded: `free` (price 0, small credit grant) + `premium` (placeholder price). `code` is the stable handle used by the API + entitlements.
- **`saas_subscriptions`** — `id, subscriber_type enum[person|tenant], subscriber_id, plan_id fk, status enum[pending|trialing|active|past_due|canceled], trial_ends_at?, current_period_start?, current_period_end?, psp_ref?, canceled_at?, timestamps`. Index `(subscriber_type, subscriber_id, status)`. Named `saas_` to disambiguate from the future tenant-plane `gym_subscriptions` (DB design §2.6).
- **`coupons`** — `id, code (unique), percent_off?|amount_off_micros?, currency?, max_redemptions?, redeemed_count int, expires_at?, active bool, timestamps`.
- **`coupon_redemptions`** — `id, coupon_id fk, subscriber_type, subscriber_id, subscription_id fk, redeemed_at`. Unique `(coupon_id, subscriber_type, subscriber_id)` (one redemption per subscriber).
- **Reused:** `ai_credit_wallets` / `ai_credit_ledger` (AiOrchestration) — already carry `plan_grant` + `period_reset_at`, anticipating this.

## 6. Lifecycle

- **Subscribe** `POST /v1/subscriptions {plan_code, coupon_code?}`
  - Free plan → activate immediately (status `active`, set period), grant `ai_credit_grant`.
  - Paid plan **with trial** → `trialing`, entitlements active, `trial_ends_at` set, and `ai_credit_grant` granted **immediately** so the trial is actually usable. The later conversion charge (webhook) continues access without a second grant for the same period; the next *renewal* period grants again (§7).
  - Paid plan **no trial** → `pending`, `createCheckout()` via gateway, return `{redirect_url, psp_ref}`. Activation deferred to webhook.
  - Coupon (if given) validated (active, not expired, under max redemptions, not already redeemed by this subscriber), price adjusted, `coupon_redemptions` row written on activation.
- **Webhook** `POST /v1/webhooks/psp/{provider}` (unauthenticated route; **authenticated by signature** via `verifyWebhook`)
  - **Idempotent** on the PSP event id (dedupe table or unique guard) — replays are no-ops.
  - `payment_succeeded` → activate subscription (`active`, set `current_period_*`, `psp_ref`), **grant `ai_credit_grant`** via `AiCreditMeter::grant(reason='plan_grant', ref=subscription)`.
  - `payment_failed` → `past_due`.
- **Upgrade / downgrade** `PATCH /v1/subscriptions/{id} {plan_code}`
  - Compute **proration**: unused portion of the current plan credited against the prorated cost of the new plan over the remaining period; settle the delta via the gateway (charge or credit). Deterministic math, fully unit-testable against `FakePaymentGateway`. Swap `plan_id`, adjust entitlements + credit grant difference.
- **Cancel** `DELETE /v1/subscriptions/{id}` (or PATCH status) → `canceled`, effective at `current_period_end` (retain access until then).
- **Top-up** `POST /v1/ai-credits/purchase {pack}` → credit-pack checkout via gateway → webhook grants the pack credits (`reason='topup'`).
- **Entitlements** — `PlanResolver::for(Person): Plan` returns the active/trialing subscription's plan, else the seeded `free` plan. A `feature.{name}` Gate ability checks the resolved plan's `features_json`; `limits_json` exposed for client + server checks. The plan's `ai_credit_grant` is the **primary** entitlement (AI usage is already metered by credits).

## 7. AICredit integration

- **Replace the stopgap:** new Persons are auto-assigned the `free` plan (on registration/onboarding), which grants its `ai_credit_grant`. The standalone `free_grant` listener is retired (or repointed to issue the free-plan subscription) so there is one funding path. Migration note: existing persons keep their granted balance; ledger reasons distinguish `free_grant` (legacy) from `plan_grant`.
- **Recurring period grants:** monthly credit refresh happens on each successful renewal webhook (`plan_grant` per period). One-time activation grant now; recurring renewal grants ride the real PSP renewal events (so fully exercised only once renewals exist — fake gateway can simulate a renewal event in tests).

## 8. Build sequence (each → its own implementation plan)

This epic is too large for one plan. Sequence:
- **A. Plans + entitlements foundation** — `plans` table + seeder, `PlanResolver`, `feature.*` Gate, `GET /v1/plans?audience=b2c`, free-plan auto-assign (retire stopgap).
- **B. Subscribe + payment seam + webhook** — `PaymentGateway` interface + `Unconfigured`/`Fake`, `saas_subscriptions`, `POST /v1/subscriptions`, `POST /v1/webhooks/psp/{provider}` (verify/idempotent/activate/grant), `POST /v1/ai-credits/purchase`.
- **C. Coupons** — `coupons` + `coupon_redemptions`, validation + redemption at subscribe.
- **D. Upgrade/downgrade + proration** — `PATCH /v1/subscriptions/{id}`, proration math, cancellation.

## 9. Testing

- **TDD throughout**, against `FakePaymentGateway` — no real PSP/network, same discipline as the AI generators against the fake `LlmGateway`.
- **Release-blocking:** credit grants are exactly-once (no double-grant on webhook replay — idempotency test), proration math (deterministic cases incl. up + down + same-day), coupon redemption limits, person-scoping on every query, webhook signature rejection.
- OpenAPI: add the 5 billing paths; run the contract guard (no CI covers it).

## 10. Non-goals / deferred

- Real PSP adapter + webhook signature scheme + concrete prices (Q2/Q3).
- Tenant (coach/gym) subscriptions — `subscriber_type` is modelled for it, but only `person` is built now (B2C).
- Recurring-renewal scheduling/dunning beyond reacting to renewal webhooks.
- Revenue analytics / super-admin billing UI (FR-SAS-005).
- Double-entry money ledger (Plane B) — AICredit ledger is single-entry signed; real-money accounting is a separate concern when revenue flows.
